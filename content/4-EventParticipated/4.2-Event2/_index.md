## What DevOps Really Does in Practice

"With great power comes great responsibility... and headaches"

The scope of DevOps work depends on context:
- Company size
- Project size
- Team structure
- Product complexity
- Platform/SRE/Security/Cloud team model
- Ownership of production support
- Existing level of automation
- Maturity of the cloud infrastructure

## What Should You Learn First?

(Reference tools: Claude, ChatGPT, Docker)

**Fundamentals**
- Linux
- Basic networking knowledge
- A programming language: Python, Golang
- Git and CI/CD
- Containers

**Understand how applications run**
- Build, test, deploy, logs, configuration, environment variables

**Start with small projects**
- Deploy a simple app, automate a task, monitor it, deliberately break it, then fix it

**Before and after AI:**
- Before: 2 hours writing code, 6 hours debugging
- After: AI generates code in 5 minutes, but debugging now takes up to 24 hours
→ Understanding the fundamentals still matters more than coding speed

## DevOps Tool Landscape

An ecosystem of tools spanning the entire lifecycle: Plan → Code → Build → Test → Release → Deploy → Operate → Monitor

12 main tool categories:
1. Planning & Collaboration
2. Source Control
3. CI/Build
4. Artifact & Package Management
5. Infrastructure as Code
6. Configuration Management
7. Containers & Orchestration
8. Cloud Platforms
9. Observability/Monitoring
10. Logging/Tracing
11. Security/DevSecOps
12. Developer Experience/Platform Engineering

Two elements that run through the entire lifecycle:
- Security/DevSecOps — security integrated at every stage
- Automation — the foundation that powers everything

## Lessons Learned the Hard Way

- Copying commands is not the same as understanding the underlying problem
- Learn to identify who is truly the owner of a problem
- Learn to ask "why" before "how"
- Communication is also part of the job
- DevOps is not about being a "hero"

## What Makes a Good DevOps Engineer?

- Tools change. Fundamentals stay.
- Stay curious and keep learning.
- Think in systems, not just individual tasks.
- Automate boring, repetitive work.
- Make things clear and easy to understand for the whole team.
- Use AI to amplify your capabilities, not to switch off your brain.

### My Experience at the Event
- Attending this meetup was very valuable and gave me a clear learning path to follow going forward.

#### Connecting and Exchanging Ideas

- The session by Trong Hoang Truong was presented in a straightforward, humorous, and very down-to-earth style, creating a comfortable atmosphere for attendees to ask questions about the DevOps learning path as well as the real challenges of the job.
- I had the chance to talk with other attendees about their experiences transitioning into DevOps, how they self-taught areas like Linux, CI/CD, and Containers, as well as real-world experiences with on-call duties and handling production incidents.

#### Lessons Learned

- Choosing to pursue DevOps shouldn't be based solely on the image of "doing CI/CD" or being "the Docker/Kubernetes person" — the actual work is much broader and depends heavily on each organization's context (company size, team structure, cloud infrastructure maturity, etc.).
- "Copying commands is not the same as understanding" — copying commands doesn't mean understanding the underlying problem; this is a very easy trap to fall into when self-learning through tutorials without digging into the fundamentals.
- "Learn to ask 'why' before 'how'" — before looking for a technical solution, you need to understand the root cause of the problem and correctly identify who actually owns it.
- "DevOps is not about being a hero" — this job isn't about one person shouldering all the responsibility to save the system whenever something breaks; it's about building processes and a culture that let the whole team operate smoothly.
- "Tools change. Fundamentals stay." — DevOps tools change constantly over time, but the fundamentals (Linux, networking, programming, systems thinking) are what truly help a DevOps engineer adapt and grow in the long run.
- AI can help you write code faster, but it can't replace understanding how the system actually works — use AI to amplify your capabilities, not to replace your own thinking.
